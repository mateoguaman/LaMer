## 2. Pipeline Structure

Each training iteration consists of **rollout collection** followed by **LLM training**.

### Rollout collection

The rollout plan is generated by `metarl_rollout_loop.py:340`:

```
(attempt=0, 'play',    4 turns)   → 4× [LLM generate → VLA inner loop (k steps)]
(attempt=1, 'reflect', 1 turn)    → 1× [LLM generate → store reflection, no VLA]
(attempt=1, 'play',    4 turns)   → 4× [LLM generate → VLA inner loop (k steps)]
```

**Totals per iteration:**
- 8 play turns, each requiring k=96 VLA inner steps
- 1 reflect turn (LLM only, ~instant on the env side)
- 2 restart calls (PyBullet `restoreState`, fast)
- Total VLA inner steps: **8 × 96 = 768** (or 8 × 128 = 1024)

### Per play-turn sequence (all sequential)

```
1. preprocess_batch()              CPU, ~10ms
   ↓
2. generate_sequences_agent()      GPUs 1-4 (vLLM, TP=2), ~1-3s
   ↓
3. envs.step(text_actions) ──TCP──→ server._handle_play_step():
                                     for step in range(k=96):
                                       vla.predict()    ← GPU 0
                                       envs.step()      ← Ray workers (CPU)
                                   ←─TCP──
   Total server time: k × (VLA + env step) per turn
```

Steps 1→2→3 are strictly sequential. The VLA inner loop (step 3) runs
entirely on the env server (GPU 0) and is the dominant cost.

### LLM training (after rollout)

```
4. compute_log_prob()              GPUs 1-4, ~3-5s
5. compute_advantage()             CPU, ~10ms
6. update_actor()                  GPUs 1-4, ~5-10s
```

## 4. Preprocessing Deep Dive

### What `vla.predict()` actually does

The VLA predict call (`lava_policy.py`) has two phases inside `_build_batch()`,
followed by a JAX model forward pass:

```
Phase 1 — Image preprocessing (_phase1_batched):
  Collect 128 RGB images from obs dicts          Python loop, dict access
  np.stack into (128, H, W, 3) uint8             115 MB contiguous copy
  [jax_gpu]: CPU→GPU transfer (uint8)            115 MB, ~5ms
  [jax_gpu]: float32 convert, crop, resize       On GPU, ~10ms
  [jax_gpu]: GPU→CPU transfer (float32)          84 MB, ~15ms
  Store preprocessed frame in per-env deque       Python assignment

Phase 2 — Batch assembly:
  Allocate (128, 4, 180, 320, 3) float32 zeros   3.5 GB allocation
  For each of 128 envs:                           Python loop
    np.stack(list(frame_buffer))                  4 frames × 216 KB = 864 KB per env
    Copy CLIP tokens (cached)                     ~0.1ms
  jnp.array(rgb_batch)                            3.5 GB CPU→GPU transfer, ~35ms
  jnp.array(clip_batch)                           Small

Forward pass:
  _forward_jit(variables, observation)            JIT-compiled JAX, ~44ms at bs=128
```

### Time breakdown at bs=128, jax_gpu mode

| Operation | Est. time (ms) | Notes |
|-----------|----------------|-------|
| Python dict unpacking + list collection | ~15 | 128 dict lookups |
| `np.stack` 128 images (115 MB) | ~10 | |
| CPU→GPU uint8 transfer (115 MB) | ~5 | |
| GPU: float convert + crop + resize | ~10 | Fast on H200 |
| GPU→CPU float32 transfer (84 MB) | ~15 | **Redundant** — goes back to CPU only to be re-uploaded in Phase 2 |
| Phase 2: Python loop, per-env `np.stack` | ~40 | 128 × 4-frame stack = 333 MB of small copies |
| Phase 2: `jnp.array` CPU→GPU (3.5 GB) | ~35 | **Redundant** — data was on GPU in Phase 1 |
| CLIP tokenization (cached) | ~3 | |
| JAX model forward pass | ~44 | Efficient, JIT-compiled |
| Output: `np.array(actions)` GPU→CPU | ~1 | Small (128 × 2 floats) |
| **Total** | **~183** | Benchmark measured: 182.8ms |

**Key finding**: ~50ms per step (~27%) is wasted on round-tripping data between
GPU and CPU. Preprocessed images go GPU→CPU (Phase 1 output) just to be stored
in frame buffers, then CPU→GPU again (Phase 2 `jnp.array`). Keeping frame
buffers on-device would eliminate this.

### Preprocess mode comparison (VLA benchmark, bs=128)

| Mode | Preprocess (ms) | Forward (ms) | Total (ms) | Throughput (samp/s) |
|------|-----------------|--------------|------------|---------------------|
| original | 365.4 | 45.0 | 410.4 | 311.9 |
| batched_tf | 171.7 | 43.7 | 215.4 | 594.3 |
| jax_gpu | 138.3 | 44.4 | 182.8 | 700.4 |

`batched_tf` is bit-exact with `original`. `jax_gpu` has max action error ~6e-4,
which is negligible for RL training.

## 5. Ray Overhead Deep Dive

### What `envs.step()` actually does

From `envs.py`, the step call dispatches to 128 Ray actors:

```python
futures = [self.workers[idx].step.remote(actions[idx]) for idx in active_indices]
results = ray.get(futures, timeout=self.timeout)
```

Each Ray worker (`LanguageTableWorker`) runs:
```python
obs, reward, done, info = self.env.step(action)      # PyBullet physics + RGB render
result_obs = self._extract_obs(obs, self.env._last_state)  # Full state dict with RGB
return result_obs, reward, done, info                 # Serialized through Ray
```

### Time breakdown at 128 envs

| Component | Est. time (ms) | Notes |
|-----------|----------------|-------|
| Create 128 Ray futures | ~2 | Python overhead |
| Ray scheduling to workers | ~5 | Task dispatch |
| Worker execution (PyBullet + render) | ~5-10 | All parallel, wall time = max |
| **Ray serialization of 128 results** | **~60** | **Each obs dict has ~922 KB RGB. 128 × 922 KB = 115 MB through Arrow/Plasma** |
| **`ray.get()` deserialization** | **~50** | **Reconstructing 128 dicts with numpy arrays** |
| Python loop unpacking results | ~15 | |
| **Total** | **~153** | Benchmark measured: 152.5ms |

**Key finding**: ~110ms per step (~72% of env step time) is spent serializing and
deserializing 115 MB of RGB images through Ray's object store. This happens
**768 times per iteration**, totaling ~84s of pure serialization overhead.

### Why this matters per inner step

The actual PyBullet physics + rendering takes ~5-10ms per step (all 128 workers
in parallel). But the Ray communication overhead adds ~140ms on top, making the
env step ~15× more expensive than the actual physics simulation.

## 6. Why Multi-GPU VLA Serving Helps

### The key insight: VLA time scales linearly with batch size

From the env_scaling benchmark, the per-environment VLA cost is nearly constant
once the batch is large enough to saturate the GPU:

| Envs | VLA (ms) | ms/env | Amortization ratio |
|------|----------|--------|--------------------|
| 4 | 38.5 | 9.62 | 1.00× (baseline) |
| 8 | 53.3 | 6.66 | 0.69× |
| 16 | 85.4 | 5.34 | 0.55× |
| 32 | 149.9 | 4.68 | 0.49× |
| 64 | 286.0 | 4.47 | 0.46× |
| 128 | 555.1 | 4.34 | 0.45× |
| 256 | 1103.3 | 4.31 | 0.45× |
| 512 | 2176.5 | 4.25 | 0.44× |
| 1024 | 4223.1 | 4.12 | 0.43× |

Above ~32 environments, VLA time scales almost perfectly linearly with batch
size — the per-environment cost plateaus at ~4.3ms. This means:

- **128 envs on 1 GPU**: VLA takes 555ms (128 × 4.34 ms/env)
- **64 envs on 1 GPU**: VLA takes 286ms (64 × 4.47 ms/env)

**If we split 128 environments across 2 GPUs (64 each), each GPU independently
runs VLA inference on its 64 environments in parallel. Since VLA time is
proportional to batch size, each GPU takes ~286ms instead of 555ms. The two GPUs
run simultaneously, so the wall-clock time for VLA inference drops from 555ms to
286ms — a 1.9× speedup.**

The same logic extends to more GPUs:

- **4 GPUs × 32 envs each**: VLA takes ~150ms per GPU → 3.7× speedup
- **6 GPUs × ~21 envs each**: VLA takes ~107ms per GPU → 5.2× speedup

The speedup is slightly less than a perfect N× because:
1. Small batches have slightly higher per-env cost (GPU not fully saturated)
2. There is a fixed overhead per VLA predict call (~20ms for JAX JIT dispatch)
3. The env step (Ray + PyBullet) also has per-dispatch overhead that doesn't
   vanish at small batch sizes

The env step portion follows a similar pattern but with more overhead at low
counts due to Ray scheduling:

| Envs | Env step (ms) | ms/env |
|------|---------------|--------|
| 32 | 55.5 | 1.73 |
| 64 | 85.7 | 1.34 |
| 128 | 152.5 | 1.19 |

### Combined effect: total step time vs number of env GPUs

Each GPU independently processes `128/N` environments. Wall-clock step time equals
the per-GPU time (all GPUs run in parallel):

| Env GPUs | Envs/GPU | VLA (ms) | Env step (ms) | **Total step (ms)** | **Throughput (env-steps/s)** | **Speedup** |
|----------|----------|----------|---------------|---------------------|------|---------|
| 1 | 128 | 555 | 153 | **708** | 181 | 1.0× |
| 2 | 64 | 286 | 86 | **372** | 344 | 1.9× |
| 3 | ~43 | 195 | 66 | **261** | 491 | 2.7× |
| 4 | 32 | 150 | 56 | **205** | 623 | 3.4× |
| 6 | ~21 | 107 | 52 | **159** | 806 | 4.5× |
| 8 | 16 | 85 | 50 | **135** | 945 | 5.2× |

With `jax_gpu` preprocessing applied (reduces VLA portion by ~2.25×):

| Env GPUs | Envs/GPU | VLA (ms) | Env step (ms) | **Total step (ms)** | **Throughput (env-steps/s)** | **Speedup vs 1-GPU original** |
|----------|----------|----------|---------------|---------------------|------|---------|
| 1 | 128 | 247 | 153 | **400** | 320 | 1.8× |
| 2 | 64 | 133 | 86 | **219** | 585 | 3.2× |
| 3 | ~43 | 103 | 66 | **169** | 760 | 4.2× |
| 4 | 32 | 79 | 56 | **135** | 952 | 5.3× |
| 6 | ~21 | 66 | 52 | **118** | 1085 | 6.0× |

### Implementation

Each additional env GPU runs its own env server process, with its own LAVA model
replica and its own subset of Ray workers. GRPO groups (8 environments per prompt)
must stay on the same server to maintain shared episode state. With 16 prompts,
the natural split is:

- 2 servers: 8 prompts × 8 = 64 envs each
- 4 servers: 4 prompts × 8 = 32 envs each
- 8 servers: 2 prompts × 8 = 16 envs each

The LaMer training client needs to be modified to shard requests across multiple
env servers rather than sending all 128 environments to a single server.

## 7. Full Node Layout Analysis

### Why training is NOT the bottleneck

From `outer_loop_results.json`, the LLM training cost per iteration:

| Phase | Config | Est. time | Notes |
|-------|--------|-----------|-------|
| LLM generation | vLLM, TP=2, 128 seqs × ~20 tokens | ~18s | 9 calls × ~2s |
| Log prob recompute | fwd only, micro_batch=32/gpu | ~3-5s | |
| Actor update | gc=on, fwd+bwd, FSDP across 4 GPUs | ~5-10s | 2 mini-batch steps × ~4.6s |
| **Total training phases** | | **~30s** | **~9% of iteration** |
| **VLA inner loop** | | **~307s** | **~91% of iteration** |

Even making training infinitely fast would only save ~30s out of ~340s. The
optimal strategy is to maximize GPUs allocated to VLA serving.

### Can more GPUs speed up training?

With the current 128 samples per iteration and `ppo_mini_batch_size=64`:

- The LLM does 2 gradient steps per epoch (128 / 64 = 2)
- Each step processes `micro_batch_per_gpu=16` per GPU
- With 4 GPUs: 4 × 16 = 64 per step → exactly 1 accumulation step per mini-batch

Adding more training GPUs would reduce per-step time (FSDP shards more), but:
- Training is already ~9% of total time
- 2× faster training saves ~15s out of ~340s (4% improvement)
- That same GPU allocated to VLA saves ~50-70s (15-20% improvement)

**Increasing batch size** doesn't help throughput either: doubling `TRAIN_NUM_ENVS`
to 32 doubles the environments to 256, but the VLA throughput at 256 envs is
roughly the same as at 128 (both ~180 env-steps/s on 1 GPU). You get 2× more
samples per iteration but each iteration takes 2× longer.

### Minimum viable training config

For Qwen3-4B with `param_offload=True` and `optimizer_offload=True`:

- Model weights: ~8 GB in bf16 (sharded across FSDP GPUs)
- vLLM requires TP=2 minimum (2 GPUs)
- FSDP across 2 GPUs with offloading is sufficient for `micro_batch=16`

**2 training GPUs is viable**, freeing up to 6 GPUs for VLA serving.

### Recommended layouts (8 GPU node)

| Layout | Env GPUs | Train GPUs | Est. VLA step (jax_gpu) | Est. iteration time | LLM samples/s | Speedup |
|--------|----------|------------|------------------------|--------------------|----|-----|
| Current (5 GPU) | 1 | 4 | 400ms | ~340s | 0.047 | 1.0× |
| Conservative | 2 | 6 | 219ms | ~195s | 0.082 | 1.7× |
| **Balanced** | **4** | **4** | **135ms** | **~130s** | **0.123** | **2.6×** |
| Aggressive | 6 | 2 | 118ms | ~120s | 0.133 | 2.8× |

**Recommended: 4 env GPUs + 4 training GPUs.** This provides 2.6× throughput
improvement while keeping the training config identical to today. The 6+2 layout
gives diminishing returns (only 8% more) and requires training config changes.

## 8. Is Training the Bottleneck?

**No.** To summarize with one number:

- VLA inner loop: **307s** (91% of iteration)
- Everything else: **33s** (9% of iteration)

Training cannot become the bottleneck unless VLA time is reduced by ~10×, which
would require both multi-GPU serving AND major per-step optimizations.

The batch size question: increasing `ppo_mini_batch_size` beyond 64 does not
increase throughput. It may improve learning efficiency (better gradient estimates)
but the rate of generating new samples is entirely gated by VLA throughput.

## 9. All Optimization Opportunities

### Tier 1: High impact

#### A. Multi-GPU VLA serving (2.6–6× throughput)

Split 128 environments across N env server processes, each on its own GPU with
its own LAVA model replica. VLA time scales linearly with batch size, so N GPUs
reduce VLA wall-clock time by ~N×. See [Section 6](#6-why-multi-gpu-vla-serving-helps)
for detailed projections.

**Estimated impact**: 2.6× at 4 env GPUs, up to 6× at 6 env GPUs (with jax_gpu).

**Implementation**: Modify slurm script to start N env servers on separate GPUs.
Modify `RemoteEnvironmentManager` to shard requests across servers. GRPO groups
(8 envs per prompt) must stay on the same server.

#### B. Eliminate GPU↔CPU round-trips in preprocessing (~15% per-step speedup)

Currently with `jax_gpu`, preprocessed images travel:
```
GPU (after resize) → CPU (frame buffer deque) → GPU (batch assembly jnp.array)
```

This wastes ~50ms per step on 128 envs. Fix: store frame buffers as JAX device
arrays. Phase 2 becomes `jnp.stack` on device instead of a Python loop +
CPU→GPU transfer.

**Estimated impact**: ~50ms saved per step × 768 steps = **38s per iteration (~11%)**.

#### C. Shared memory for Ray worker observations (~17% per-step speedup)

Each inner step serializes 128 RGB images (115 MB) through Ray's object store.
This costs ~110ms per step. Replace with pre-allocated shared memory
(`multiprocessing.shared_memory`): workers write RGB directly, main process
reads without serialization.

**Estimated impact**: ~100ms saved per step × 768 steps = **77s per iteration (~22%)**.

#### D. Fused JIT: preprocess + frame buffer + forward (~25% per-step speedup)

Currently, only the model forward pass (`_forward_fn`) is JIT-compiled in
`lava_policy.py`. The preprocessing pipeline (`_jax_preprocess_rgb_batch`)
and the batch assembly loop (Phase 2 in `_build_batch`) run as eager
Python/NumPy code, causing multiple GPU↔CPU round-trips and Python loop
overhead every step.

**What's currently JIT-compiled** (`lava_policy.py:134`):

```python
self._forward_jit = jax.jit(self._forward_fn)
```

This covers only the ResNet forward pass + action denormalization (~44ms).

**What's NOT JIT-compiled:**

- `_jax_preprocess_rgb_batch`: crop + resize on GPU, but returns `np.asarray()`
  (GPU→CPU transfer) so the result can be stored in CPU-side frame buffers
- Phase 2 of `_build_batch`: Python for-loop over 128 envs calling `np.stack`
  per frame buffer, then `jnp.array` to re-upload the 3.5 GB batch to GPU
- Frame buffer management: Python `collections.deque` holding CPU numpy arrays

**Why JIT-ing preprocess alone is modest (~5–8ms):**

The GPU crop + resize kernel is already fast (~10ms at bs=128). Wrapping it
in `jax.jit` saves only JAX dispatch overhead. The real cost is the
**GPU→CPU transfer afterward** (`np.asarray()`, ~15ms) and the **CPU→GPU
re-upload in Phase 2** (`jnp.array()`, ~35ms). These transfers exist because
frame buffers live on CPU — JIT-ing preprocess in isolation doesn't change that.

**The big win: fuse the entire predict path into one JIT call.**

Merge preprocessing, on-device frame buffer update, and forward pass into a
single `jax.jit`-compiled function. Frame buffers become persistent JAX device
arrays that never leave the GPU:

```python
@functools.partial(jax.jit, static_argnums=(3,))
def _fused_predict(variables, frame_buffers, images_uint8, clip_tokens, active_mask):
    """
    frame_buffers: (num_envs, history_len, H, W, 3) float32, on device
    images_uint8:  (num_envs, H_raw, W_raw, 3) uint8, transferred once per step
    """
    # 1. Preprocess on device (replaces _jax_preprocess_rgb_batch)
    images = preprocess_on_device(images_uint8)          # (N, 180, 320, 3)

    # 2. Update frame buffers on device (replaces Python deque + np.stack)
    frame_buffers = jnp.roll(frame_buffers, shift=-1, axis=1)
    frame_buffers = frame_buffers.at[:, -1].set(images)

    # 3. Build observation dict on device (replaces Phase 2 Python loop)
    obs = {"rgb": frame_buffers, "instruction_tokenized_clip": clip_tokens}

    # 4. Forward pass (already JIT-compiled, now fused into same trace)
    actions = forward_fn(variables, obs)

    return actions, frame_buffers
```

**What this eliminates per step (at 128 envs):**

| Eliminated operation | Savings (ms) | Why it's eliminated |
|---------------------|-------------|---------------------|
| GPU→CPU after preprocess (`np.asarray`) | ~15 | Frame buffers stay on device |
| Phase 2 Python loop (128× `np.stack`) | ~40 | Replaced by `jnp.roll` + `.at[].set()` |
| CPU→GPU batch assembly (3.5 GB `jnp.array`) | ~35 | Data never left GPU |
| Multiple JAX dispatch/sync points | ~23 | Single fused kernel launch |
| **Total** | **~113** | |

**Estimated impact**: ~113ms saved per step × 768 steps = **87s per iteration (~25%)**.

If implemented, this subsumes optimization B (on-device frame buffers is a
natural consequence of the fused JIT) and optimization G (the fused function
includes its own on-device preprocessing). B and G are listed separately since
they are also useful as standalone incremental steps.

**Practical considerations:**

- Frame buffers must be initialized as `jnp.zeros` on device at episode start
  and at each environment reset
- `active_mask` must be handled inside the JIT via `jnp.where` (no Python `if`)
- CLIP tokenization must remain outside the JIT (string ops aren't jittable),
  but tokens can be cached on-device since goals don't change within a turn
- The raw `uint8` CPU→GPU transfer (~5ms for 115 MB at 128 envs) remains —
  camera images must cross from CPU to GPU once — but this is the only
  remaining host↔device transfer per step
- First call incurs XLA compilation (~2–5s), amortized over thousands of steps

### Tier 2: Medium impact

#### D. Batch Ray dispatch

Replace 128 individual `worker.step.remote()` calls with a single coordinator
actor that receives all actions and dispatches internally. Reduces Ray
scheduling overhead from 128 futures to 1.

**Estimated impact**: ~5-10ms per step.

#### E. Pre-allocate contiguous frame buffers

Replace per-env `deque` of 4 frames with a single `(num_envs, 4, H, W, 3)`
array. Phase 2 batch assembly becomes a slice instead of 128 `np.stack` calls.

**Estimated impact**: ~30ms per step at 128 envs.

#### F. Preprocess on workers before Ray transfer

Have each Ray worker resize/crop RGB to 180×320 before returning. Reduces
per-image transfer from 922 KB to 216 KB (128 envs: 115 MB → 27 MB).

**Estimated impact**: ~40ms per step (reduced serialization). Tradeoff: ~1ms
per worker (parallel), saves ~40ms on main process.

#### G. Use jax_gpu as default preprocess mode

Already planned. Gives 2.25× speedup on VLA predict vs original mode, confirmed
bit-exact up to ~6e-4 max action error (negligible for RL).

### Tier 3: Lower priority

#### H. vLLM tuning

With TP=2 and `gpu_memory_utilization=0.6`, there's room to increase KV cache
or enable chunked prefill. But LLM generation is ~5% of total time; a 2×
speedup here saves ~9s.

#### I. Reduce observation serialization to essentials

Workers currently return full state dicts. Stripping to only the keys needed
(rgb, reward, done) reduces serialization per worker.

### Combined projection

If A (4 env GPUs) + C (shared memory) + D (fused JIT) are all implemented:

| Optimization | Per-step savings | Notes |
|-------------|------------------|-------|
| Fused JIT (D) | ~113ms from VLA predict | Subsumes B and G; includes on-device preprocessing + buffer management |
| Shared memory (C) | ~100ms from env step | Eliminates 115 MB Ray serialization |
| 4 env GPUs (A) | ~3.4× from batch splitting | 128→32 envs per GPU |

**Step 1 — Fused JIT alone (128 envs, 1 GPU):**

```
VLA predict:  247ms - 113ms = ~134ms
Env step:     153ms (unchanged)
Total:        ~287ms per step  (28% faster than jax_gpu baseline)
```

**Step 2 — Add shared memory (128 envs, 1 GPU):**

```
VLA predict:  ~134ms
Env step:     153ms - 100ms = ~53ms
Total:        ~187ms per step  (53% faster than jax_gpu baseline)
```

**Step 3 — Add 4 env GPUs (32 envs/GPU):**

At 32 envs per GPU, fused JIT savings scale down (smaller data transfers):

```
VLA predict:  79ms (jax_gpu at bs=32) - ~30ms (fused JIT at bs=32) = ~49ms
Env step:     56ms (bs=32 baseline) - ~25ms (shared memory at bs=32) = ~31ms
Total:        ~80ms per step
```

**Projected throughput with all three:**

```
768 steps × 80ms = 61s VLA time
+ 18s LLM gen + 10s training + 5s overhead = ~94s per iteration
16 samples / 94s ≈ 0.17 samples/s
```

**4.9× improvement** over current 0.035 samples/s (original mode, 1 env GPU),
or **3.6× improvement** over 0.047 samples/s (jax_gpu mode, 1 env GPU).

## 10. Realistic Benchmark Design

### What's missing from current benchmarks

The existing benchmarks (`benchmark_envs.py`, `benchmark_vla.py`,
`benchmark_outer_loop.py`) measure components in isolation. They miss:

1. **End-to-end iteration timing** including LLM generation and TCP round-trips
2. **Interaction between phases** (e.g., GPU memory contention between vLLM and
   FSDP on shared GPUs)
3. **Actual n, m, k parameters** (benchmarks used k=20, not k=96)
4. **Steady-state behavior** across multiple iterations (JIT warmup, cache effects)

### Proposed benchmark: `benchmark_end_to_end.py`

A standalone script that runs the actual code path for a controlled number of
iterations, measuring every phase:

**Server-side instrumentation** (in `lava_env_manager.py` and `lava_policy.py`):
- Per inner step: VLA preprocess time, VLA forward time, Ray dispatch time,
  Ray collect time, total step time
- Per play turn: total inner loop time, number of active envs over time
- Return timing data as metadata in the TCP response

**Client-side measurements**:
- Per turn: preprocess_batch, LLM generation, TCP round-trip (including
  server-side execution), total turn time
- Per iteration: rollout total, log_prob recompute, advantage compute,
  actor update, total iteration time

**Configuration**:
- Uses actual env server (started by the benchmark)
- Uses actual vLLM engine with real model weights
- Runs with production n, m, k values
- Executes 3-5 full iterations for steady-state measurement

**Output**: JSON timing trace + summary table with per-phase breakdown,
enabling direct comparison when testing optimizations.
